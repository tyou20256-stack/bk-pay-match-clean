## 設計判断の記録

### DJ-001: フレームワークなしのフロントエンド
**判断:** React/Vue等を使わず、Vanilla HTML/CSS/JSで構築
**理由:**
- ビルドプロセスが不要（HTMLファイルを直接配信）
- 依存関係が少なく、保守が簡単
- 画面数が少ない（5画面）ため、SPAのメリットが小さい
**トレードオフ:** コンポーネント再利用が困難。画面が増える場合はReactへの移行を検討。

### DJ-002: SQLiteの選択
**判断:** PostgreSQL/MySQLではなくSQLiteを使用
**理由:**
- インストール不要（Node.jsライブラリのみ）
- 単一ファイルでバックアップが容易
- 同時接続数が少ない（管理者1-2名 + 顧客の注文API）
- WALモードで読み取りパフォーマンスは十分
**トレードオフ:** 高頻度の同時書き込みには不向き。1日の注文数が1000を超える場合はPostgreSQLへの移行を検討。

### DJ-003: P2P API は広告一覧のみ
**判断:** P2P注文作成はPuppeteer（ブラウザ自動化）で行う
**理由:**
- 4取引所（Bybit/Binance/OKX/HTX）全てにP2P注文作成の公開APIが存在しない
- 利用可能なのは広告一覧（オーダーブック）の取得のみ
- 実際の注文作成にはブラウザでのログインが必要
**調査結果:** 各取引所のAPI docsを確認。order/create系エンドポイントはスポット/先物取引のみで、P2P/OTCには非対応。

### DJ-004: HTXの無効化
**判断:** HTX P2P APIのcurrency=11を使用しない
**理由:**
- currency=11がJPY（日本円）ではなくRUB（ロシアルーブル）であることが判明
- 全通貨ID（1-30）をスキャンし、JPYに該当するIDがないことを確認
- 支払方法もSberbank/Tinkoff等のロシアの銀行のみ
**証拠:** 88円のレートはUSDT/RUBレートとして正確。支払方法がAlipay/WeChat（中国系）とロシア銀行のみ。

### DJ-005: Cookie認証
**判断:** JWTではなくセッションCookieを使用
**理由:**
- サーバーサイドでセッション無効化が容易（DBから削除するだけ）
- HttpOnly Cookieでクライアント側JSからアクセス不可（XSS対策）
- 管理画面はブラウザからのアクセスのみなので、Cookieが自然
**トレードオフ:** スケールアウト時にセッション共有が必要。現段階では単一サーバーなので問題なし。

### DJ-006: SHA-256パスワードハッシュ
**判断:** 初期実装はSHA-256 + 固定ソルト
**理由:**
- 外部ライブラリ不要で即実装
- MVP段階で速度優先
**リスク:** ブルートフォースに弱い。本番ではbcrypt/argon2への移行を強く推奨。
**TODO:** Phase 2でbcryptに移行。

### DJ-007: AUTO + SELF の二段構え
**判断:** P2Pマッチングを試み、失敗時に自社口座にフォールバック
**理由:**
- P2P取引所で最安レートを取得したい（コスト最適化）
- しかし常にマッチングが成功するとは限らない（金額/支払方法/オンライン状況）
- 自社口座のフォールバックがないと、顧客の入金が処理できなくなる
**ユーザー判断:** ミスターKKが「どの道どちらも必要」と明言。

### DJ-008: 30秒更新間隔
**判断:** レートを30秒ごとに更新
**理由:**
- P2P市場はスポット市場ほど変動が激しくない
- 3取引所×3通貨×買い売り = 18 APIリクエスト/更新
- CoinGecko無料枠（月10,000回）の制約
- 10秒は過剰、60秒は遅い → 30秒が妥当
**トレードオフ:** 急激なレート変動時に最大30秒の遅延。

### DJ-009: 15分の注文タイムアウト
**判断:** 注文作成から15分で自動キャンセル
**理由:**
- P2P取引所のマーチャントも一般的に15-30分のタイムアウト
- 短すぎると顧客が振込を完了できない
- 長すぎるとレート変動リスクが増大
- 15分は銀行振込に十分な時間

### DJ-010: 乖離率フィルターのデフォルト15%
**判断:** スポットレートから±15%超のオーダーを除外
**理由:**
- P2Pプレミアムは通常5-10%程度
- 15%超は異常値（詐欺/入力ミス/流動性の低い通貨ペア）
- ダッシュボードのUIで動的に調整可能（1-50%）
- サーバー側でも適用しAPIレスポンスを軽量化

## 試みて失敗/断念したこと

### F-001: Cloudflare Quick Tunnels
**状況:** ngrokの代替として試行
**結果:** `invalid UUID length: 0` エラーで接続不可
**対処:** ngrokに変更（アカウント作成が必要だが安定）

### F-002: HTX JPY P2P
**状況:** 4番目の取引所として追加
**結果:** currency=11がRUBであることが判明（レートが半分近い値）
**対処:** 無効化。3取引所（Bybit/Binance/OKX）で運用

### F-003: P2P注文作成API
**状況:** 4取引所の公開APIでP2P注文を自動作成しようとした
**結果:** 全取引所でP2P注文作成APIが非公開
**対処:** Puppeteerブラウザ自動化フレームワークを構築

### F-004: localtunnel
**状況:** ngrokの代替として試行
**結果:** 動作はするが「Click to Continue」画面が毎回表示され、UX悪い
**対処:** ngrokに変更

### F-005: Puppeteerスクリーンショット
**状況:** ドキュメント用の画面キャプチャをPuppeteerで自動取得
**結果:** Protocol error / Connection closed でクラッシュ（Node.js v25との互換性問題の可能性）
**対処:** ASCII図とテキストベースの画面説明で代替
